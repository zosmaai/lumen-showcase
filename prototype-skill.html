Prototype

Throwaway code that answers one design question — then gets deleted

Midnight Editorial · 2026

01

The Core Idea

Prototype builds a small, disposable program to answer one design question. The code is throwaway from day one — no tests, no error handling, no abstractions.

The point: learn something fast, then delete it. The moment you start hardening prototype code, you've stopped prototyping.

  • One command to run
  • State in memory only
  • Full state surfaced on every step
  • Marked clearly as prototype, not production
Code is disposable. The answer is the only thing worth keeping.

The prototype discipline

When the prototype answers its question, capture the verdict somewhere durable — commit message, ADR, issue, NOTES.md — then delete or absorb the code.

02

Two Branches — Question Decides

🔧 Logic Prototype

"Does this state model feel right?"

  • Interactive terminal app
  • Push state machine through edge cases
  • Full state printed after every action
  • Portable logic module behind thin TUI shell
  • Validated reducer can be lifted into real code

🎨 UI Prototype

"What should this look like?"

  • 3 radically different UI variations
  • Switchable via ?variant= URL param
  • Floating bottom bar for navigation
  • Butt up against real page with real data
  • Structurally different — not color tweaks
03

Five Rules — Apply to Both Branches

#RuleWhy
1 Throwaway day one — marked as prototype Context is obvious; no accidental production merge
2 One command to run — existing task runner User starts without thinking
3 No persistence default — in-memory state Persistence is what you might be checking, not something to depend on
4 Skip polish — no tests, no error handling, no abstractions Point is to learn fast, then delete
5 Surface state — print full state on every step User sees what changed, spots the design bugs

Logic Prototype — The Shape

// PROTOTYPE — throwaway TUI // Question: Does cart state machine handle partial refunds? // ← PORTABLE MODULE — lift this into real code type CartState = { items: Item[]; refunds: Refund[] }; function applyRefund(state, refund): CartState { ··· } // ← TUI SHELL — deleted after prototype function main() { let state = initialCart(); loop { console.clear(); renderState(state); // pretty-printed renderKeys(); // [r] refund, [q] quit key = readKey(); state = dispatch(state, key); } }

Rule: TUI imports the logic module. Nothing flows the other direction.

06

UI Prototype — Sub-Shapes

Two sub-shapes, one preferred:

  • Sub-shape A (preferred) — Adjust an existing page. Variants on the same route via ?variant=. Real data, real components. Design problems surface when you butt up against reality.
  • Sub-shape B (last resort) — Brand new throwaway route at /prototype/<name>. Only when there's genuinely no existing page to host variants. Empty routes hide design problems populated ones would expose.

Default to 3 variants. Cap at 5. More becomes noise, not signal.

08

Anti-Patterns — What Kills a Prototype

Logic Branch

  • Adding tests — if it needs tests, it's no longer prototype
  • Wiring real database — question is about design, not persistence
  • Generalizing — "what if we supported X later"
  • Blurring logic and TUI — console.log in the reducer
  • Shipping TUI into production — delete shell, lift module

UI Branch

  • Variants differ only in color — not a prototype, wallpaper
  • Sharing too much code — shared <Layout> defeats the point
  • Wiring real mutations — question is "what should look like"
  • Promoting prototype directly to production — rewrite properly
  • New page when existing one would work — empty routes hide problems

10

Prototype as a Wayfinder Ticket Type

In the Wayfinder ecosystem, prototype is one of four ticket types. Unlike research (fully AFK), prototype is HITL — Human In The Loop. You need a human to drive the artifact and react.

  • Research — AFK: agent reads, writes cited Markdown
  • Prototype — HITL: build cheap artifact, human reacts
  • Grilling — HITL: interview, human decides
  • Task — HITL or AFK: manual work, result recorded

Prototype answers feed to-spec and domain-modeling. Validated direction becomes settled input for the spec.

Build Cheap. Learn Fast. Delete Fearlessly.
