Components

Four subsystems

Each boundary already has a name in the code — the two Next.js planes, the LiteLLM data plane, the Redis counter, and Postgres.

1 · Control plane (Next.js + Better Auth)

Better Auth handles Google OAuth and writes the standard auth tables via a Drizzle adapter. Org creation is disabled client-side — orgs are created server-side during provisioning instead. The control plane is the only caller that holds the LiteLLM master key.

Key modules:

src/auth.tslib/session.tslib/provisioning.tslib/litellm.tslib/org.tslib/crypto.tsapp/api/*

2 · Data plane (LiteLLM proxy)

Pinned ghcr.io/berriai/litellm:main-stable. Exposes one customer-facing model qwen-4b, mapped to upstream openai/qwen3.5-4b on devserver:8001 (llama-swap behind nginx). LiteLLM enforces per-key model access and records spend/requests into its own Postgres tables. Quota enforcement is delegated to the callback, registered as callbacks: [callbacks.flat_cost.proxy_handler_instance].

3 · Quota enforcement (Redis atomic counter)

The core reliability decision. flat_cost.py runs on every proxy request and counts against reqcount:{user_id}:{YYYYMMDD} (UTC) using atomic INCR — exact and lag-free even under bursts, unlike LiteLLM's batch-flushed daily table (~10s lag) which leaks past the cap under load.

  • First increment of the day sets EXPIREAT next UTC midnight → auto-reset.
  • Over-cap request is DECR'd back and rejected with 429 (BudgetExceededError).
  • Genuine upstream failure refunds the slot; our own cap rejection is excluded.
  • Redis unreachable → fail open (never block traffic on an infra hiccup).

Reset is a date rollover, not a timer — nothing runs at midnight. A new UTC day reads a fresh key that starts at 0; the old key expires on its own TTL.

4 · Persistence (Postgres)

One Postgres instance, two databases: litellm (LiteLLM's managed keys/spend/daily tables) and app (application tables via Drizzle). Raw API keys are never stored — only an AES-256-GCM encrypted_key for reveal and a masked_key for display. litellm_user_id equals the Better Auth user id, so one id threads session → plan → key → quota counter.

Flows

Three cross-subsystem paths

Provisioning on first load, inference under the quota, and the dashboard usage read.

Provisioning (first authenticated load)

ensureProvisioned() runs from the dashboard layout on every load. It short-circuits if already provisioned and retries cleanly if a prior attempt half-failed — each step is idempotent.

Inference + quota enforcement

The customer calls the proxy directly with their own key. The callback enforces the cap before the request reaches the model.

Dashboard usage read

The dashboard polls /api/usage every 5s (and on window focus). Redis is the source of truth; LiteLLM's daily table is the fallback so an infra hiccup doesn't zero the display.

Decisions

Why it's built this way

The choices that shaped the architecture, with the alternative each one rejected.

Redis INCR over LiteLLM's daily-activity table for enforcement

Alternative: read LiteLLM's own per-user daily count. Rejected because that table is batch-flushed (~10s) — a burst all reads the same stale count and leaks past the cap. The atomic counter is exact under concurrency. LiteLLM's table is still written as historical record, just not read for enforcement.

Enforce in a LiteLLM callback, not Next.js middleware

Alternative: gate requests in Next middleware. Rejected because the customer hits the proxy directly, so the cap has to live there — and the Redis v4 client can't run on Next's Edge runtime anyway.

Fail open on Redis down

Alternative: fail closed (block when the counter is unreachable). Rejected because a quota-store outage should degrade to "unmetered", never to "all traffic blocked".

Idempotent provisioning on every load, not a one-shot signup webhook

Alternative: provision once at signup. Rejected because a half-failed attempt (org created, key generation failed) would need manual repair. Re-running each load with "already exists" caught everywhere self-heals.

Keys encrypted at rest (AES-256-GCM), never stored raw

Alternative: store the raw key for easy reveal. Rejected because a DB leak would expose live credentials. Reveal decrypts on demand; the dashboard only shows the masked form.

litellm_user_id == Better Auth user id

Alternative: a mapping table between auth users and LiteLLM users. Rejected because reusing the id threads session → plan → key → quota counter with zero extra joins.
