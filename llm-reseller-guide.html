LLM API Reseller — Build Guide
Architecture · Build Guide · 2026-07-14

Resell LLM access without building auth, keys, or limits

Put LiteLLM Proxy in front of real providers. Customers self-signup with Better Auth, pay with Razorpay, and get a metered API key. LiteLLM enforces budget + rate limits. You write ~4 endpoints of glue.

The idea in one strip
Customer→ Your app (Better Auth + Razorpay)→ LiteLLM Proxy→ OpenAI / Anthropic

Your app is the control plane (signup, billing, key minting). LiteLLM is the data plane — customers send their sk-... key straight to it, you stay out of the latency path.

Core decision
Model each customer as a LiteLLM user, not just a key
Budget lives on the user, survives key rotation, aggregates across a customer's keys (prevents budget-bypass), and gives one clean /user/info view. Cost: one extra API call at signup. Worth it.
Stack: Next.js + Postgres + LiteLLM + Redis Region: India (Razorpay, INR) Identity key: Better Auth user.id == LiteLLM user_id

CriticalKeep the two Postgres databases separate

LiteLLM runs its own Prisma migrations on boot. Don't share its DB with Better Auth — schema collisions and upgrade pain. Two DBs, one Redis.

NoteRedis is mandatory the moment you run >1 LiteLLM replica

Without it, each replica counts rate limits locally and customers get more throughput than you sold.

Data-plane vs control-plane

Your app never proxies LLM traffic. It only calls LiteLLM's admin API (with the master key) to mint/adjust keys. Customer's LLM requests go directly to the LiteLLM proxy URL using their own sk-... key. This keeps your app off the hot path and means an app outage doesn't stop paying customers from calling models.

Better Auth — self-hosted, you own the users

Unlike a hosted SaaS, Better Auth is a library in your Next.js app. No external signup webhook — user provisioning happens in-process via a database hook. That's simpler: the LiteLLM call runs right after the user row commits.

Adapter: Drizzle/Prisma → Postgres A Session: cookie (default) Tables it creates: user, session, account, verification
Config + provisioning hook
// auth.ts
export const auth = betterAuth({
  database: drizzleAdapter(db, { provider: "pg" }),
  emailAndPassword: { enabled: true, requireEmailVerification: true },
  socialProviders: { google: { clientId: ..., clientSecret: ... } },
  databaseHooks: {
    user: {
      create: {
        after: async (user) => {
          // provision LiteLLM user (free tier) right after signup
          await litellm("/user/new", {
            user_id: user.id,          // reuse Better Auth id verbatim
            user_email: user.email,
            max_budget: 1.0,           // free tier ($)
            budget_duration: "30d",
            models: ["gpt-4o-mini"],   // cheap-only until they pay
            rpm_limit: 20,
          });
          await db.insert(customers).values({
            userId: user.id, litellmUserId: user.id,
            tier: "free", litellmProvisioned: true,
          });
        },
      },
    },
  },
});
DesignGate free-tier provisioning behind verified email

Public signup + free credit = bot bait. Either provision lazily on first verified login / first "generate key" click, or keep free-tier models to the cheapest one only. Make /user/new idempotent so retries are safe.

Why store credit_balance_inr if LiteLLM is the source of truth?

Only to render "₹ credits left" in your own currency without an API call per page load. The real budget of record is LiteLLM's max_budget (USD, priced against provider costs). If you don't need INR display, drop the column and read straight from /user/info.

Razorpay — two models
Recommended
Prepaid credits
Customer buys ₹X → Order → Checkout → payment.captured webhook → raise max_budget. Set budget_duration: null for a lifetime balance. No proration, cashflow-positive.
Optional
Subscription
Razorpay Plans + Subscriptions. Each subscription.charged → set max_budget + budget_duration: "30d". Predictable revenue; you eat overage risk.

TrustBump budget in the webhook, not the client redirect

The browser callback can be dropped (tab closed). Use client-verify only for instant UI feedback; do the real budget change on payment.captured.

  1. Signup (Better Auth) → databaseHooks.user.create.after → POST /user/new (free tier).
  2. Generate key — dashboard button → your server route → POST /key/generate → show sk-... once.
  3. Buy credits — Razorpay Order → Checkout → payment.captured webhook.
  4. Raise budget — verify sig → idempotency check → POST /user/update (bump max_budget).
  5. Use — customer calls LiteLLM proxy with sk-...; budget + rate enforced automatically.
  6. Dashboard — GET /user/info → spend + remaining budget.

Key generation route (server-side, holds master key) 
// app/api/keys/generate/route.ts
export async function POST() {
  const session = await auth.api.getSession({ headers: await headers() });
  if (!session?.user) return new Response("Unauthorized", { status: 401 });

  await ensureLiteLLMUser(session.user.id, session.user.email); // idempotent
  const { key } = await litellm("/key/generate", { user_id: session.user.id });
  return Response.json({ key });   // shown once, never stored raw
}
UXNever store the raw sk-... key

Show it once. LiteLLM keeps the hash. Lost key → "rotate" button = /key/delete + /key/generate. Budget survives because it lives on the user.

Razorpay gotchas
CriticalTwo secrets: KEY_SECRET ≠ WEBHOOK_SECRET

Client-return signature uses KEY_SECRET; webhook body HMAC uses WEBHOOK_SECRET. Mixing them is the #1 Razorpay bug.

CriticalWebhook HMAC is over the RAW body

If your framework parses JSON before you hash, the signature won't match. Capture raw bytes in the webhook route.

MoneyINR paise + auto-capture

Amounts are integer paise (₹100 = 10000). Set payment_capture: 1 or money sits authorized-not-captured. Convert INR→USD budget with margin; re-check rate periodically.

MoneySubscription cycle ≠ LiteLLM reset window

Razorpay bills on its schedule; LiteLLM budget_duration resets from budget creation time. On each subscription.charged, explicitly re-set the budget instead of trusting the two to stay aligned.

SetupTest vs live keys

Separate key pairs and webhook secrets. Verify you flipped to live before launch — test keys never move real money.

Better Auth & LiteLLM gotchas
SecurityMaster key stays server-side, forever

It's root over your whole LLM spend. Only your API routes/server actions touch it. Never in client components, never exposed by Better Auth.

ReliabilityProvisioning gap → reconciliation

If /user/new fails in the hook, a Better Auth user exists with no LiteLLM account. Mark litellm_provisioned:false and run a small retry job (cron or on-login) over false rows. Requires idempotent /user/new.

RoutesWebhook route is public — guard by signature, not session

Razorpay calls it unauthenticated. Verify HMAC only. Don't require a Better Auth session there (and don't leave dashboard routes session-less).

OpsPin LiteLLM to main-stable

Admin API moves fast. A floating main-latest tag can break your integration overnight. Back up Postgres B before upgrades (boot-time migrations).

MoneyCost estimate ≠ real provider bill

LiteLLM prices from its internal map; budget checks are per-request (a single call can nudge slightly over). Reconcile against your real OpenAI/Anthropic invoice monthly; price customers with margin + headroom.
