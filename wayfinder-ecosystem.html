The Wayfinder Ecosystem

How to ship quality software with AI coding agents — without losing your mind

The Problem

Raw agents produce chaos

AI coding agents are incredibly fast. But speed without process amplifies entropy:

  • Alignment gaps — you think the agent knows what you want. It doesn't.
  • Misdirection — agents charge ahead, answer their own questions, and build the wrong thing.
  • Codebase rot — velocity accelerates software entropy. Ball-of-mud arrives in weeks, not months.
  • No shared language — every session starts from zero. No glossary, no decisions captured.

More speed × zero process = faster failure.

The Answer

Process before code

The Wayfinder ecosystem is a suite of composable agent skills that enforce engineering fundamentals at every stage:

Wayfinder

A shared map on your issue tracker

Wayfinder plans work too big for one agent session. It creates a single map issue with child investigation tickets, wired together with native blocking relationships.

  • Destination first — naming the goal fixes scope and shapes every ticket
  • Plan, don't do — each ticket resolves a decision; map is done when the route is clear
  • Fog of war — deliberately incomplete; resolving one ticket clears fog, graduating new tickets
  • Claim by assignment — assignee is the claim. No labels needed.

The First Gate

Grilling: killing misalignment dead

The #1 failure mode: you think the dev knows what you want. Then you see they built. Same thing happens with AI agents — magnified.

  • One question at a time — never bewilder with a list
  • Recommended answer — each question comes with a suggestion
  • Facts vs decisions — look up facts in the codebase; ask humans for decisions
  • Walk the design tree — resolve every branch before building anything

Without grilling, every other skill is just faster wrong.

❌ Without process

"Build the thing" → agent guesses → wrong thing built → rewrite → context lost → repeat

No spec, no tickets, no shared understanding. Every session starts from scratch.

✅ With process

Grill → Spec (problem, stories, decisions) → Tickets (vertical slices with blockers) → Implement one at a time

Spec lives on the issue tracker. Tickets declare blocking edges. CI stays green batch to batch.

Key Concept

Vertical slices beat horizontal layers

Each ticket cuts a narrow but complete path through every layer — schema, API, UI, tests. A completed slice is demoable on its own.

  • Sized to fit in a single context window
  • Tracer bullets — thin slice first, then widen
  • Wide refactors use expand→contract: add new form beside old, migrate in batches, delete old

Horizontal slices (all schema, then all API, then all UI) never produce something you can test.

Foundation

Domain Modeling: shared language

A shared vocabulary makes everything more concise and navigable:

  • CONTEXT.md — living glossary at the repo root. Pure terms, no implementation.
  • ADRs — Architectural Decision Records. Only when: hard to reverse, surprising, real trade-off.
  • Challenge fuzzy language — "You said 'account' — did you mean Customer or User?"
  • Stress-test with scenarios — force precision at the boundaries

Consistent naming → easier navigation → fewer tokens wasted on disambiguation.

Quality Gate

Two-axis code review

Every change passes through two independent lenses:

Standards

Does the code follow this repo's conventions, patterns, and style?

Spec

Does the code match what the originating issue asked for?

One can pass while the other fails. Reporting separately prevents masking.

The Case

Why this beats going raw

Each skill solves a real, expensive failure mode:

  • Misalignment → Grilling catches it before code is written
  • Verbosity → Shared language compresses communication
  • Bugs → TDD at pre-agreed seams + diagnosing loop
  • Codebase rot → Architecture reviews every few days
  • Huge projects → Wayfinder maps the path without doing the work

Software engineering fundamentals matter more in the AI age — not less.
