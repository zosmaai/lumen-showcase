Why Navigator?

The Matt ecosystem's /implement skill used subagents for TDD. Navigator replaces them with interactive branching.

Core idea: Instead of spinning up parallel subagents that run isolated prompts, the LLM push-tasks a prompt into the session tree. You drive the branch — you steer the work, ask clarifying questions, and cancel dead ends. Each branch collapses into a summary that folds back into the main conversation.

The push-task Bridge

The LLM calls push-task({ prompt: "..." }) to pre-queue work. This stores a task in the session tree — nothing happens until you run /start-branch or /start-fresh. Multiple tasks stack; the nearest pending task fires first.

push-task(seam) → /start-branch → TDD loop → /return (summary)

The TDD Pipeline

Step-by-step implementation flow with navigator branching.

  1. Break the spec into seams
    The LLM identifies pre-agreed seams from the spec or ticket. Each seam is a self-contained unit of work.
  2. Push each seam as a task
    push-task({ title: "TDD: auth middleware", prompt: "Write failing test for auth middleware, implement to pass, run typecheck + tests" })
  3. Instruct the user
    "Run /start-branch to work on auth middleware. Run /return when done."
  4. Steer the branch
    You're in an interactive TDD loop. You can stop mid-branch, re-prompt, cancel dead ends, or return early.
  5. Branch returns
    /return collapses the branch into a summary. The next /start-branch auto-loads the next queued task.
  6. Full test suite + review
    All seams done → run full suite → /code-review → commit.

The stacked pipeline is sequential, not parallel. Each branch must complete before the next loads. This is intentional — it gives you full control over the order and lets you abort between seams.

When Navigator Wins

Spike investigations — Branch off, explore a library's API, return with findings. You control the depth.
Fresh-eyes reviews — /start-fresh gives the LLM zero conversation bias. No prior context to anchor to.
Interactive TDD — Test failures, design pivots, and test prioritization benefit from human steering between iterations.

When Subagents Still Win

Independent parallel work — 2-5 unrelated tasks that don't need iteration. Example: /code-review's Standards + Spec axes can run in parallel.

How push-task stacking works

Multiple push-task calls queue tasks in the session tree (LIFO — nearest fires first). The branch command searches backward from the current leaf, finds the nearest pending task, and injects its prompt as the first message.

After /return, a task-done marker is injected, preventing re-fire. /cancel also marks as done (silently), so dead ends don't re-appear.

Branch summaries and context compression

When you /return, Pi compresses the entire branch conversation into a single summary entry on the main branch. The main LLM reads this summary and continues — it gets the findings without the back-and-forth cluttering context.

This means you can do extensive exploration in a branch and it costs only one summary line on the main conversation.

Integration with other Matt skills

Wayfinder — Navigator branching is the natural execution engine for Wayfinder tickets. Each ticket becomes a branch.

Triage — /start-fresh is perfect for triage reviews: fresh eyes on an issue without conversation bias.

TDD — Each seam's TDD loop runs inside a branch, with full interactivity.

Code Review — Keep subagents for parallel Standards+Spec review. Navigator for the TDD implementation.

Common Patterns

Pattern 1: Single Seam
One push-task → /start-branch → TDD → /return. Simplest form.
Pattern 2: Pipeline
Multiple push-task → chain of branches. Each /start-branch loads the next task automatically.
Pattern 3: Fresh Review
push-task → /start-fresh (clean context) → review → /return. Zero bias review of specs or designs.
The key insight: navigator branching gives you full context at fraction of the cost compared to subagents. You trade parallelism for quality, control, and cheaper execution.
